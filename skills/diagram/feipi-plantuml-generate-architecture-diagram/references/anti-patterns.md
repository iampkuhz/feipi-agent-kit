# 常见反模式

## 先画图，后补需求

- 表现：用户只有一句抽象目标，就直接开始写 PlantUML。
- 问题：组件、边界和流程会被模型脑补，后续很难核对责任边界。
- 修复：先补 `architecture-brief`，至少写清层、组件、流程三件事。

## 把模板、样例、规则各写一份

- 表现：模板注释写一套规则，说明文档再写一套，脚本里还有第三套。
- 问题：规则一改就漂，维护成本迅速失控。
- 修复：schema 作为唯一真相，模板只示例，文档只解释。

## 组件名和 alias 不一致

- 表现：brief 写 `order_service`，图里却用了 `orderService` 或 `orders`.
- 问题：覆盖校验无法自动化，图和需求无法机械比对。
- 修复：alias 只来自 `components[].id`，不要重新命名。

## 只看源码不看渲染

- 表现：`.puml` 看起来没问题，就直接交付。
- 问题：真实渲染时常见重叠、过宽、标签遮挡。
- 修复：至少执行渲染脚本；若环境不可用，明确标注为未完成真实渲染校验。
