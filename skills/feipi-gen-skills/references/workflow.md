# 工作流与迭代

## 强约束原则
- 简洁优先：`SKILL.md` 只保留高价值信息，正文建议 <= 500 行，超出拆分到 `references/`。
- 验证优先：先定义验收标准再实现，无可执行验证证据视为未完成。
- 自由度匹配风险：高自由度（多方案）、中自由度（可参数化）、低自由度（高风险需按序）。
- 渐进式披露：`SKILL.md` 提供导航与流程，细节放 `references/` 并由 `SKILL.md` 直接一跳链接。
- 先读现状再动手：优先从当前对话、现有 skill、失败样例中抽取信息，避免重复追问用户。
- 迭代基于证据：先看漏触发、误触发、失败输出和维护成本，再决定改 `description`、正文还是脚本。

## 四阶段工作流（Explore -> Plan -> Implement -> Verify）
- Explore：收集任务目标、触发时机、输入输出、边界与风险；只读必要文件，先索引后定向打开。
- Plan：明确改动文件、理由、验证方法、是否需要旧版本对照；变更可一句话描述 diff 时可直做，不做重规划。
- Implement：先落可复用资源（脚本/参考/资产），再完善 `SKILL.md`；确定性操作优先脚本化；避免 Windows 路径。
- Verify：运行 `make validate DIR=<skill-root>/<name>`，并完成至少一种任务级验证；交付给出验证步骤、结果、剩余风险。

## 反馈循环
- 默认采用循环：验证 -> 修复 -> 再验证。
- 如果问题主要是“不会触发/乱触发”，优先重写 `description`。
- 如果问题主要是“会触发但做不好”，优先修正文流程、模板、脚本或参考资料。
- 如果问题主要是“维护成本高”，优先重构目录分层与脚本入口。

## 高风险任务处理
- 触发条件：批量改动、破坏性操作、复杂规则。
- 步骤 1：先生成中间计划文件（如 `changes.json`）。
- 步骤 2：用脚本校验计划。
- 步骤 3：通过后再执行变更。

## 测试与迭代要求
- 至少准备 3 个评估场景（正常/边界/异常）。
- 优先真实任务回放，不只做静态阅读。
- 若目标环境涉及多模型，至少在预期模型档位做一次对照测试。
- 根据观察到的失败行为迭代，不基于猜测优化。

## 触发描述优化
- `description` 要同时覆盖“做什么”和“什么时候用”，重点写用户意图，不写实现细节堆砌。
- 为避免漏触发，可适度主动描述相关语境，但不要罗列过长关键词清单。
- 优化描述时至少检查两类案例：
  - 漏触发：明明该用却没用。
  - 误触发：用户意图相近但其实不该调用。
- 若只有描述问题，不要顺手重写整份 `SKILL.md`；保持最小改动。

## 用户协作方式
- 先复用用户已提供的信息，再补问真正缺失项。
- 面向不熟悉术语的用户时，把 `eval`、`baseline`、`assertion` 等词翻成中文目的说明。
- 默认给一个推荐方案；只有在目录、命名或风险有明显分叉时再停下来确认。
- 展示测试样例时，优先用真实用户口吻，而不是命令式假数据。

## 反模式与修复
- 说明冗长且重复常识：删解释，保留流程、约束、命令与示例。
- 给太多并列方案导致选择困难：给一个默认方案 + 一个例外逃生舱。
- 只给规则，不给验证：补充可执行验证步骤与通过条件。
- 路径/目录组织混乱：用语义化文件名，按领域拆分 `references/`。
- 过度依赖环境变量控制：优先改为脚本常量或自动探测；仅在必须使用外部凭据时保留最少环境变量。
