# 专利创新交底书生成（中文）

## 核心目标

输入业务与技术信息，输出一份结构完整、内容具体、可校验的《创新提案交底书》Markdown 文档。

## 触发场景

1. 用户要把零散创新点整理成专利前置材料、发明交底初稿或可评审交底书。
2. 用户已经给了业务背景、方案步骤、模块信息，但需要按固定章节重组为正式文档。
3. 用户要求同时产出图示、控制章节篇幅，并在交付前自动检查格式约束。
4. 用户希望先生成完整交底书，再据此继续扩写权利要求或评审材料。

## 非适用场景

1. 只做一句话创意润色，不需要形成完整交底书。
2. 只要求生成单张技术图，不需要完整文档结构。
3. 缺少核心方案信息且用户明确不愿补充，导致无法形成可核对内容。
4. 需要治理、重构或迁移 skill 本身时；这类任务应使用 `feipi-skill-govern`。

## 先确认什么

开始撰写前必须先核对以下输入，并区分“必填”和“可选”：

1. 必填
- 专利名：用于标题 `# 一种XXX方法/系统`。
- 使用场景：用于申请说明、背景、详细流程中的场景细节。

2. 强烈建议补齐
- 业务背景与现状痛点。
- 本方案核心做法：模块、步骤、判定逻辑、关键阈值或关键链路。
- 预期技术效果或相对现有方案的改进点。

3. 可选
- 应用本方案的产品/项目（含时间）。

规则：
1. 缺少任意必填项，必须先提问并等待确认，禁止自行假设填充。
2. 可选项未提供时不再追问，在对应章节写“暂无明确产品/项目（计划在该场景验证，时间待定）”或“无”。
3. 若核心方案只有抽象口号，没有模块、步骤或场景事实，先要求用户补 3 至 5 条可核对事实，再进入写作。

## 图示协同（强约束）

图示必须依赖当前仓库中已治理完成的 PlantUML skills：
1. 模块图、分层图优先使用 `feipi-plantuml-generate-architecture-diagram` 生成并校验。
2. 主流程图优先使用 `feipi-plantuml-generate-sequence-diagram` 生成并校验。
3. `详细描述`章节至少 1 图，建议 2 图（模块图 + 主流程图）。
4. 图代码必须是 PlantUML 代码块（`@startuml` ... `@enduml`）。
5. 流程图中的序号（如 `S1`、`S2`）必须与正文流程序号一致。
6. 模块图必须分层（2-3 层，优先 3 层），且不同一级层使用可区分颜色。
7. 主流程图的核心步骤控制在 5-10 步；若使用时序图，所有参与方（participant/actor/user 等）控制在 5-7 个。
8. 核心创新图（如提供）必须比“详细描述”的图更精简，只保留最关键对象与链路。

协作边界：
1. 本 skill 负责决定“要画什么图、图和正文如何对齐、图后如何解释”。
2. `feipi-plantuml-generate-architecture-diagram` 与 `feipi-plantuml-generate-sequence-diagram` 负责生成并校验 PlantUML 代码与渲染结果。
3. 若图示未通过对应 diagram skill 的校验或可读性差，不得硬贴进交底书；先修图再继续写作。
4. 当前仓库没有通用活动图 skill；若用户只接受活动图，必须先明确说明当前默认以时序图替代，不能假装已有通用绘图器。

## 输入与输出

1. 最少输入
- 专利名
- 使用场景
- 业务背景与现状痛点
- 本方案核心做法（模块/步骤）
- 应用产品/项目（含时间，可选）

2. 输出
- 一份 Markdown 交底书，标题从 `# 一种XXX方法/系统` 开始。
- 标题顺序、标题名称与模板一致。
- 图示若存在，必须能追溯到对应 diagram skill 的校验结果。
- 通过 `scripts/check_disclosure_format.sh` 格式校验。

## 执行流程（使用态）

1. Explore
- 先判断用户处于哪一阶段：只有想法、已有提纲、已有完整素材待成稿。
- 优先从当前对话中提取已知信息，不重复追问已经给出的内容。
- 缺少必填项或关键事实时，先提问补齐，不直接开写。

2. Plan
- 按模板把素材映射到各章节，先列章节提纲，再决定图示方案。
- 先规划图示与正文共用的序号体系（`S1...Sn`），避免图文错位。
- 明确哪些内容写事实，哪些内容写判断，避免“先下结论再补理由”。

3. Implement
- 先用模板搭出完整骨架，再逐章填充，避免漏章节。
- `申请说明`保持简洁；`背景`写事实锚点；`详细描述`写成一条最常见场景下的完整流程。
- 图示先生成 brief，再分别交给对应 diagram skill 生成并校验，最后插回正文。

4. Verify
- 对产物执行格式校验：
```bash
bash scripts/check_disclosure_format.sh <交底书.md>
```
- 若失败，优先修复缺章、字数、图文序号不一致等硬错误，再检查表达质量。
- 若图示来自外部协作，必须确认相关 diagram skill 已完成自身校验，不能只凭肉眼判断。

5. Iterate
- 若结果过于空泛，优先补事实锚点，而不是堆修饰词。
- 若结果图文不一致，优先统一 `S1...Sn` 编号与步骤名称。
- 若章节失衡，优先压缩“背景/术语”，把篇幅让给“详细描述/创新点”。

## 固定排版模板（强约束）

必须使用：`assets/proposal_template.md`

规则：
1. 不改标题层级，不改标题文本。
2. 不删除章节标题；如无内容可写“无”。
3. 可在章节正文内补充四级标题、列表、图示，但不能改变主模板结构。

## 编写重点

### 1. 先写事实，再写判断
- 禁止套话、大话、空泛原则，必须落到具体问题、具体步骤、明确场景。
- 每条核心结论优先写全“对象 + 动作 + 条件/结果”三要素，至少覆盖其中 3 项。
- 判断句前必须出现可核对事实，如时间、阈值、步骤序号、模块名、数据现象。

### 2. 控制章节职责
- `申请说明`必须是 `背景` 的简化版，只写核心问题、方案动作、解决结果。
- `术语解释`直接解释该术语在本方案中的含义；若与业界常见定义不同，在句末补一句“与传统差异”。
- `是否还有其他解决方案`写“本方案变体”，不是无关替代品，最多 2 条。
- `技术效果总结` 与 `关键技术创新点`统一格式：
  - `[序号] **摘要短语**：一小段详细描述`

### 3. 把核心篇幅留给“详细描述”
- `详细描述`按固定顺序写：
  - 一段话讲清技术本质与核心价值。
  - 给出模块图并解释模块职责。
  - 给出主流程图，正文流程序号与图中序号一致。
  - 展开核心创新点或核心模型，可配更精简的创新图。
- `详细描述`建议控制在 1000 至 1500 字区间，重点写“最常见场景的一条完整流程”。
- 主流程图下方文字优先写“核心步骤摘要”，避免与图逐句重复；图无法表达的细节再补充。

### 4. 保持工程复盘口吻
- 优先一层列表（`1.` / `-`），减少长段落与并列大话。
- 控制“首先/其次/最后/此外”等总结腔连接词密度，避免连续堆叠。
- 句式偏短，少用形容词，优先使用可执行动作词，如采集、判定、路由、回执、回滚、补救。

## 标准命令

1. 校验单个交底书
```bash
bash scripts/check_disclosure_format.sh <交底书.md>
```

## 验收标准

1. 模板章节完整，顺序正确。
2. 字数、条目数量、格式约束满足校验脚本。
3. `详细描述`章节包含图示时，图示应与正文序号一致，且能追溯到 diagram skill 的校验结果。
4. 核心创新点、技术效果、相似方案缺点均能从正文中找到事实支撑，而不是纯口号总结。

## 常见失败与修复

1. 文档看起来完整，但内容空泛
- 处理：回到输入阶段，补充真实场景、模块名、阈值、失败环节、前后差异。

2. 图示能渲染，但和正文步骤对不上
- 处理：统一 `S1...Sn` 编号、步骤名称和模块边界，再重新生成或改图。

3. `申请说明`、`背景`、`详细描述`互相重复
- 处理：收紧 `申请说明`，把背景留给事实锚点，把流程与创新细节集中到 `详细描述`。

4. “其他解决方案”写成泛泛替代方案
- 处理：改写为本方案变体，只保留仍能达成近似效果的 1 至 2 条。
