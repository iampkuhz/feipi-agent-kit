# 示例

## 示例 1：架构图页

### 用户输入

```
我们做了一个本地多 agent 调度架构。
包含以下组件：
- Codex：负责代码生成和审查
- Claude Code：负责交互式开发辅助
- Qoder：负责需求分析和任务分解
- LiteLLM：作为模型路由层，统一调度不同模型
- 任务队列：Redis 实现的异步任务分发
- 审查器：安全策略检查，拦截不合规请求

数据流：用户请求 → Qoder 分解 → 任务队列 → LiteLLM 路由 → Codex/Claude Code 执行 → 审查器 → 返回结果

核心瓶颈：LiteLLM 路由层的延迟占整体响应时间的 40%。
```

### 信息充足性判断

- 主题：本地多 agent 调度架构 ✓
- 结论：LiteLLM 路由延迟是核心瓶颈 ✓
- 原始事实：6 个组件及其职责、完整数据流 ✓
- 关系：顺序流程 + 路由分发 ✓

四项齐全，信息充足。

## 示例 2：对比矩阵页

### 用户输入

```
对比三种公链建设路线：

1. 串行共识路线
- 代表：传统 EVM 链
- 优点：生态成熟、工具链完善
- 缺点：TPS 瓶颈明显，扩容靠 L2
- 适合：对性能要求不高、重视生态的项目

2. 并行共识路线
- 代表：Solana 类链
- 优点：原生高 TPS
- 缺点：开发门槛高、生态相对封闭
- 适合：高吞吐场景、自建技术团队

3. Multi-proposer 路线
- 代表：新兴多提案者架构
- 优点：去中心化程度高
- 缺点：共识复杂度高、目前缺乏大规模验证
- 适合：重视去中心化、愿意承担技术风险的项目
```

### 信息充足性判断

- 主题：三种公链建设路线对比 ✓
- 结论：各路线适合不同场景，无绝对优劣 ✓
- 原始事实：3 个对象 × 4 个维度（代表、优点、缺点、适合场景）✓
- 关系：并列对比 ✓

四项齐全，信息充足。

### Page Contract 示例

```
## Page Contract

### 1. 本页目标
面向 CTO，对比三种公链建设路线的优劣和适用场景，支撑技术路线决策。

### 2. 一句话结论
三条路线无绝对优劣，选择取决于项目优先级：生态成熟度 vs 吞吐量 vs 去中心化程度。

### 3. 使用的原始信息
- 三条路线：串行共识、并行共识、Multi-proposer
- 每条路线的代表、优点、缺点、适合场景（用户提供）

### 4. 页面内容范围
本页包含：
- 3 列对比表格（路线 × 维度）
- 顶部推荐倾向标注

本页不包含：
- 具体 TPS 数值（用户未提供，不自造）
- 代码行数 / 开发周期估算（用户未提供）
- 市场判断或投资建议

### 5. 推荐页面结构
- 主表：comparison-matrix 版式，3 列 × 4 行（代表、优点、缺点、适合场景）
- 顶部 Highlight：标注"无绝对推荐，取决于优先级"
- 结论区：Bottom bar 写取舍结论

### 6. 生成前确认
请确认是否按这个内容范围生成当前页 PPT。
```

### 生成策略说明

- 版式选择 `comparison-matrix`，因为内容是多方案对比。
- 用户没有提供 TPS、LOC、开发周期等具体数值，**绝对不自造**。
- 表格维度使用用户提供的四个维度：代表、优点、缺点、适合场景。

## 示例 3：路线图页

### 用户输入

```
自建高性能公链三阶段规划：

第一阶段（0-6 个月）：基础设施
- 目标：搭建共识层和网络层
- 关键事项：PoS 共识实现、P2P 网络搭建、基础虚拟机开发
- 交付：测试网上线，TPS > 1000

第二阶段（6-12 个月）：生态工具
- 目标：构建开发者工具和基础 DApp
- 关键事项：SDK 发布、浏览器钱包、DEX 部署
- 交付：主网上线，日活地址 > 1000

第三阶段（12-24 个月）：规模化
- 目标：提升性能和生态多样性
- 关键事项：并行执行优化、跨链桥、DeFi/NFT 生态扩展
- 交付：TPS > 10000，生态项目 > 100

主要风险：第二阶段生态冷启动困难，需要激励机制吸引开发者。
```

### 信息充足性判断

- 主题：自建高性能公链三阶段规划 ✓
- 结论：三个阶段递进推进，第二阶段是生态冷启动风险点 ✓
- 原始事实：3 个阶段，每阶段有目标、事项、交付 ✓
- 关系：阶段递进 ✓

四项齐全，信息充足。
