# 交互协议

## 交互目标

- **收敛**：每次交互都指向一个明确目标——确认 Page Contract、生成 PPT、或补充原始信息。
- **不问不必要问题**：默认假设已知，只问缺少的关键事实。
- **不让用户选风格**：风格、受众、颜色、密度全部内置默认值。
- **不把 skill 变成调研 agent**：不负责外部搜索、不负责补充未经用户提供的事实。
- **不把 skill 变成需求访谈机器人**：不引导用户做需求澄清，只做信息充足性检查。

## 默认交互流程

```
一页一交互
一页一确认
先检查信息
再生成 Page Contract
确认后才生成 PPT
```

每一页独立处理，不批量生成（除非用户明确要求多页规划）。

## 追问规则

最多问 **3 个阻塞问题**。超过 3 个仍未补齐关键信息，停止追问并说明当前能做什么、不能做什么。

### 只问原始信息，不问偏好

**可以问：**

- 这一页想表达的核心结论是什么？
- 需要展示的对象/模块/阶段/方案有哪些？
- 每个对象有哪些事实、数据、特征或关系？
- 这些对象之间是流程、分层、对比、因果还是阶段关系？

**不要问：**

- 你喜欢什么风格？
- 你想给谁看？
- 要不要科技风？
- 要不要蓝色？
- 要不要高密度？
- 要不要加图标？

这些全部使用默认值。

## Fast Mode

如果用户明确说出以下任一表述：

- "不要确认，直接生成"
- "跳过 Page Contract"
- "你自己处理"
- "快速生成"

则可以进入 **fast mode**，跳过 Page Contract 确认环节，直接生成 PPT。

但即使 fast mode，也必须：

1. 先做信息充足性检查——信息不足时仍要要求补充。
2. 不做外部调研。
3. 不编造事实。
4. 不添加未经用户提供的数据、指标、结论。

Fast mode 只是跳过确认环节，不是跳过所有规则。

## 多页处理

如果用户要求一次处理多页：

1. 先输出多页规划（每页一句话主题）。
2. 逐页执行单页流程：信息检查 → Page Contract → 确认 → 生成。
3. 不要一次性生成全部 PPTX 而不做中间确认。

## 语言

默认使用中文回复。技术术语、缩写可以保留英文（CTO、PPTX、Page Contract、Visual QA、Architecture Map 等）。
