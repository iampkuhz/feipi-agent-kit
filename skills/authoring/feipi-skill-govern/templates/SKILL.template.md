# {{TITLE}}

## 核心目标

- 用一句话写清这个 skill 要稳定完成什么事。
- 输出应可验证，而不是只给一段看起来合理的文字。

## 适用场景

- 列出典型触发语句或任务场景。
- 写清输入类型、预期输出和默认策略。

## 不适用场景

- 写清容易混淆但其实不该调用本 skill 的场景。
- 若用户缺少关键输入，也在这里写明先补什么，不直接执行。

## 先确认什么

1. 必填
- 用户必须提供的输入。

2. 按需确认
- 只有在特定分支下才需要补问的输入。

## 工作流（Explore -> Plan -> Implement -> Verify）

1. Explore：仅读取必要文件，避免无边界探索。
2. Plan：先定义改动范围与验收标准，再实现。
3. Implement：优先复用 `scripts/`、`references/`、`assets/`。
4. Verify：执行当前任务产物校验并记录结果。

## 输入与输出

1. 输入
- 写清用户需要提供什么。

2. 输出
- 写清最终要交付什么，以及如何让结果可核对。

## 标准命令

- 只放完成当前任务所需的命令，不要放仓库维护命令。

## 验收标准

1. 至少有一种可执行校验方式（脚本、命令输出、截图比对）。
2. 输出必须包含“验证步骤 + 验证结果 + 剩余风险”。
3. 若验证失败，先修复再交付。

## 资源说明

- `scripts/`：放可复用、确定性强的辅助脚本。
- `references/`：放按需读取的领域资料或长规则。
- `assets/`：放输出时要用到的模板或静态文件。

## 环境变量约定（如需）

- 新增环境变量前，先确认能否通过参数或自动探测替代。
- 若必须新增：变量名与同类 skill 保持一致，不自行发明同义变量名。
- 环境变量模板统一维护在仓库根目录 `.env.example`，不要在 skill 下新增 `references/.env.example`。
