Field Definitions & Formulas

All derived metrics are computed in Python (index/metrics.py). When a denominator is 0 or data is missing, the UI shows "—" instead of 0.

Input Tokens

Tokens sent to the model as input (prompt). Split into fresh, cache read, and cache write categories.

Metric Definition Formula Anthropic Field OpenAI Field
Input Fresh New tokens that were not cached. The actual "new" input cost. input_tokens input_tokens input_tokens - cached_tokens
Cache Read Input tokens served from cache (read hit). Cheaper than fresh input. cached_input_tokens cache_read_input_tokens cached_tokens (prompt)
Cache Write Input tokens written to cache this turn. One-time cost for future reads. cached_output_tokens cache_creation_input_tokens not reported
Input-side Total Sum of all input-side token categories. fresh + cache_read + cache_write Sum of 3 fields Sum (cache_write excluded)
Output Tokens

Tokens generated by the model as output (completion).

Metric Definition Formula Anthropic Field OpenAI Field
Output Visible Tokens visible in the chat output (not reasoning/thinking). output_tokens output_tokens completion_tokens - reasoning_tokens
Reasoning Tokens used for internal reasoning/thinking. — not separated reasoning_tokens
Derived Metrics & Formulas

These metrics are computed from raw token fields. All are defined in index/metrics.py. Division by zero returns — (None).

Metric Formula Interpretation
Cache Reuse Ratio cache_read / (input_fresh + cache_read + cache_write) How much input is served from cache. Higher = more efficient. 0% means no cache hits.
Cache Write Ratio cache_write / (input_fresh + cache_read + cache_write) Fraction of input being written to cache. High values indicate active cache building.
Output Ratio output / (input_fresh + cache_read + cache_write) Output-to-input ratio. Very low values (<5%) with large input may indicate context bloat or idle rounds.
Tools/Round tool_call_count / assistant_message_count Average tool calls per conversation round. High values may indicate complex tasks or tool loops.
Tokens/Round (input_side_total + output) / assistant_message_count Average tokens consumed per round. High values suggest large context windows.
Tokens/Minute (input_side_total + output) / (duration_seconds / 60) Token throughput rate. Very low values with long duration may indicate idle processes.
Output/Minute output / (duration_seconds / 60) Output generation rate. Useful for comparing session efficiency.
Token Precision

How accurately the token count is reported.

Precision Meaning When Used
provider-reported Exact count from the provider's API. Most accurate. Anthropic, OpenAI, qwen-anthropic-compatible
estimated Derived from available fields. May not match provider's internal count. When only total tokens available (Codex)

Common Pitfalls

Cache read ≠ output cache

cache_read_input_tokens are input-side cache hits (reading from cache). cache_creation_input_tokens are also input-side (writing to cache). Neither refers to output caching.

OpenAI does not report cache write

OpenAI's API does not expose a cache_creation_input_tokens-equivalent field. The "Cache W" column for OpenAI sessions shows not reported, not 0. Do not assume zero cache write — the provider simply does not report it.

Don't double-count cache tokens

cache_read_input_tokens is NOT part of input_tokens. They are separate fields. Total input = input_tokens + cache_read_input_tokens + cache_creation_input_tokens.

— means unknown, not 0

When a token field is missing or the precision is unknown, the UI shows — instead of 0. This prevents misleading "zero token" impressions.

Estimated ≠ provider-reported

When only total tokens are available (e.g., Codex), we distribute them evenly across rounds. These are estimates, not exact counts from the provider.

Known Limitations
# Limitation Impact
1 Provider token口径差异 Different providers use different token counting methods. Anthropic uses cl100k-based encoding; OpenAI uses its own tokenizer. Cross-provider comparisons should be approximate.
2 Missing cache fields Some providers (e.g., OpenAI) do not return cache write fields. Sessions from these providers will show cache reuse ratio based on cache read only, and cache write as "not reported".
3 Partial parse sessions Some sessions may be incompletely parsed due to log corruption or format changes. These sessions are flagged with the parse_partial anomaly type. Token counts may be underestimated.
4 Duration accuracy Duration is computed from log timestamps and may be affected by: process suspension/resume, laptop sleep/hibernation, log rotation gaps, and timezone inconsistencies. Very long durations (e.g., 17h+) may include idle time rather than active processing.
5 Codex token estimation Codex only reports tokens_used (total). Per-round breakdown is estimated by distributing evenly across rounds. This is an approximation and does not reflect actual per-round variation.
6 Anomaly thresholds are heuristic Anomaly detection uses P90/P95 percentiles with a 20-session minimum. Below that threshold, fixed fallback values are used. Thresholds may not fit all workflows — adjust FALLBACK_THRESHOLDS in index/percentiles.py if needed.
